- What a decision tree is
- That more than one tree can be constructed for a given dataset

=== Launch
@slidebreak{Launch}

Consider each of the three situations described below:

- You and your friends take an online quiz that tries to guess your hairstyles.
- Two people who work for the _same company_ and earn the _same salary_ apply for a loan by filling out the _same online form_ at the local bank. One of them is granted a loan, but the other is not.
- A news anchor displays a map with some states colored blue and others colored red. She explains that the map's predictions for the upcoming election were developed using historical census data.
- A camera at the bank captures a picture of your face, in an attempt to match the picture to your identity.

What do all of these examples have in common?

@slidebreak{Launch}

On the surface, these situations may not seem to have a lot in common. But all of these are examples of @vocab{classifiers}, a kind of function that:

@indented{
(1) Receives input

(2) Asks questions about that input

(3) Use the answers to produce a predicted outcome from a list of categories
}

@QandA{
@Q{What data do you think is used for *guessing your hairstyle*?}
@A{Viewing history, search history, channel subscriptions, likes, dislikes, google activity, "not interested" feedback}

@Q{What data do you think is used for *approving loan requests*?}
@A{Debt, assets, payment history, number of open accounts, delinquent accounts, income/revenue, expenses, overdrafts in the last 90 days}

@Q{What data do you think is used to *predict election outcomes*?}
@A{Census data including voter race, gender, income level, party affiliation, and age, as well as prior years' results}
}

For example, the hairstyle quiz will _probably_ rule out "ponytail" if the sex of the person taking the quiz is "male", but it won't always be right: After all, some people with ponytails are male! A classifier's job to guess the *most likely option* based on the @vocab{training data}.

=== Investigate
@slidebreak{Investigate}

@vocab{Classifiers} use something called a @vocab{Decision Tree}, asking questions in order to narrow down the possible outcome targets until it reaches a single, predicted outcome.

If you've ever played the game "I spy", you've already got some practice building decision trees! The rules are pretty simple:

- A player thinks of something they can see, that the other plays can also see
- They come up with a hint, to help narrow the possible answers. For example: "I spy with my little eye...something blue!"
- The other players ask yes-or-no questions to help them guess what it is

==== Decision Tree Terminology

@right{@image{images/terminology-tree.png, 300}}

--
- A @vocab{decision node} splits a dataset around the values in one of its columns. The attribute in the column serves as the "question" that is being asked.
   * The very top decision node is called the @vocab{root node}. It represents an entire dataset.
- A @vocab{leaf node} is a node that contains the predicted outcome. These outcomes come from a designated column in the original dataset. Leaf nodes have no outgoing branches.
- @vocab{Splitting} is the process of creating branches and additional nodes corresponding to subsets of a dataset based on values in one column. 
  * (We're sticking to splits that only create two branch nodes, but some decisions trees use splits that create more than two.)
  * (Computer scientists call the sequence of questions from the root node to an outcome a path.)
--

@lesson-point{A @vocab{decision tree} is one way to model the structure hidden in the data.}

Unlike humans, who can generate their own questions, machine-learning algorithms generate decision trees from @vocab{training datasets}, using the column headers as questions. Each node in the tree splits the dataset, allowing us to make the best prediction we can about some outcome.

Creating a decision tree is a form of @vocab{supervised learning}, because the training data already contains the desired outcomes (as one column), and the algorithm just learns a function that maps from input to outcomes.

=== Synthesize
@slidebreak{Synthesize}

You've learned that some @vocab{decision trees} are better than others. For example, the _frequency of each category_ affects the "best node" ("if everyone has glasses...").

When choosing a split for our root node, our goal is to ask a question that sorts the data into the _neatest, most organized piles that we can_. For example, it would be ideal to get all the cats on one side, and all the dogs on the other. We can't always get a perfect split like this, but we want to get as close as possible.

@QandA{
@Q{How do you know if a question is a good one to start with or not?}
@A{Good questions help us eliminate many of the remaining categories whether the answer is yes _or_ no}

@Q{What do we want to be true about each question we ask?}
@A{We would want it to cleanly split the list of possible targets as evenly as possible, to guarantee eliminating a significant number of options.}

@Q{How does someone use a decision tree to make predictions?}
@A{Ask the question at the root of the tree, then follow the branches to ask additional questions until we get to a category at the end of a path.}

@Q{Suppose we were playing "Classify", but now you have to guess which *country* I'm thinking of. What might be a good starting question?}
@A{Answers will vary, but look for answers that eliminate groups of countries quickly.}
@A{Is it a country in the Western Hemisphere?}
@A{Is it a country where people speak Spanish?}
@A{etc...}
}

== Building Decision Trees in Pyret

=== Launch
@slidebreak{Launch}

Let's learn how to write a @vocab{classifier function} to predict species in Pyret!

Remember:

- A function consumes some input and produces some output.
- Each input will only ever have one output. 
   * If our classifier is good, it will _always_ predict that `firepaw` is a cat.
   * If it accidentally classifies him as a snail, it will _always_ predict that he's a snail.

We'll need to choose a question. Then we'll need to write a rule.

@Q{What species should our rule predict for the "yes" side of the tree?}
@A{`"cat"` - it is the most common.}

@Q{What species should our rule predict for the "no" side of the tree?}
@A{`"snail"` - snail and tarantula are equally common, but we still have to pick one of them.}

```
# tail-classifier :: Row -> String

fun tail-classifier(r):

    "cat"
  else:

  end
```

@slidebreak{Investigate}
You've probably noticed that:

- Just like any other function - `tail-classifier` has a contract! It consumes a Row (an animal) and produces a String (the species that the classifier predicts the animal will be).
- `tail-classifier` is using a row-accessor to access the information in the `"tail"` column.
- This function uses some code we haven't seen before like `if` and `else`.

@slidebreak{Launch}
@QandA{
@Q{What does this part of the function definition do `if r["tail"] == true: "cat"`?}
@A{It checks to see whether the value in the `"tail"` column is `true`, and, if it is, it classifies the animal as a `"cat"`.}

@Q{What does this part of the function definition do `else: "snail"`?}
@A{It checks to see whether the value in the `"tail"` column is `true`, and, if it says anything other than `true`, it classifies the animal as a `"snail"`.}
}

@teacher{
You might find it duplicative to check whether or not `r["tail"] == true` since the value in the cell itself is either `true` or `false`... and you're right! 

- Our program would work just as well if it just read `r["tail"]`. 
- But, using `r["tail"] == true` is consistent with how we use row accessors with non-Boolean columns. 
- When the cell values aren't `true` or `false` we do have to check whether they match the value we have in mind, e.g. `r["species"] == "cat"`. 

We have learned over time that many students do better with keeping things consistent.
}
